# Flexibility & Shifting

How flex events, earnings, and the shift simulator work.

## FlexEvent

A demand-side flexibility event ‚Äî a window where the grid operator incentivises
households to reduce or shift their usage.

```ts
interface FlexEvent {
  id: string;
  startTs: number;       // start of window (Unix-ms, half-open [start, end))
  endTs: number;         // end of window (Unix-ms)
  label?: string;        // e.g. "turn down"
  category: FlexCategory; // 'use-less' | 'use-more' | 'other'
  pricePerKwh?: number;  // incentive price in ¬£/kWh
  minFlexKwh?: number;
  maxFlexKwh?: number;
}
```

Events are loaded from `/data/flexibility_opportunity.json`. Time-only
start/end values (e.g. `"18:00"`) are expanded into one instance per day
within the price-data range. Each event's `category` is resolved once at
intake by `classifyFlexEvent()`. Only events whose (half-open) window overlaps
the usage day are shown in Flex Insights (`windowOverlapsRange`).

## Flex earnings

`computeFlexEarnings` sums a household's kWh within each event window, clamps
to `[minFlexKwh, maxFlexKwh]`, and computes earnings as
`shiftableKwh √ó pricePerKwh √ó 100` (¬£ ‚Üí pence). Events without `pricePerKwh`
are skipped.

## Event classification

`classifyFlexEvent()` resolves each event's `category` at intake, from the
label:

- **"turn down"** / **"reduce"** ‚Üí `use-less`
- **"turn up"** / **"increase"** ‚Üí `use-more`
- Everything else ‚Üí `other` (general flex opportunity)

`InsightCardList` switches on `event.category` for icon, verb, and sort order ‚Äî
it no longer re-matches the label. A daily cost comparison card always appears
first.

## Shift Simulator

Models the cost impact of moving energy between time periods.

### Time groups

The day is split into four 6-hour periods (12 half-hour slots each):

| Period | Range | Slots |
|--------|-------|-------|
| Morning | 06:00‚Äì12:00 | 12‚Äì23 |
| Afternoon | 12:00‚Äì18:00 | 24‚Äì35 |
| Peak | 18:00‚Äì00:00 | 36‚Äì47 |
| Night | 00:00‚Äì06:00 | 0‚Äì11 |

### simulateShift

```
simulateShift(usage, prices, household, fromSlots, toSlots, kwhToShift) ‚Üí ShiftResult
```

1. Strips any overlapping timestamps (shifting within the same slot is a no-op).
2. Removes energy from the from-period **proportionally** to each slot's share.
3. Distributes the shifted energy **evenly** across to-period slots.
4. Returns `{ originalCostPence, newCostPence, savingPence }`.

Positive `savingPence` = saves money. Negative = costs more.

### Settlement day & view-model

The simulator runs on a `SettlementDay` (`settlementDay.ts`) ‚Äî usage + prices
bound to one midnight-anchored grid that owns the time-group slot windows and a
`fullyPriced` coverage check, and exposes `groupUsage` / `simulateGroupShift` /
`dailyCost`. All button tones, slider bounds, and the outcome are computed by
the pure `shiftSimulatorView()` over that day; the React component holds only
the selection state and renders the resulting view-model.

### UI behaviour

The simulator has three states driven by `fromIdx` and `toIdx`:

- **Idle** ‚Äî no selection. All buttons show the household colour as outlines.
- **One selected** ‚Äî the selected button is filled (`soft`). The opposite row
  shows colour-coded hints: green outline = saves money, red outline = costs
  more, greyed out = negligible or same period.
- **Complete** ‚Äî both selected. Buttons show the outcome colour. Slider is
  enabled and defaults to the maximum shiftable amount. The same period is
  disabled on the opposite row.
