# Flexibility & Shifting

How flex events, earnings computation, and the shift simulator work.

## FlexEvent

A demand-side flexibility event ‚Äî a window where the grid operator or supplier
incentivises households to **reduce or shift** their electricity usage.

```ts
interface FlexEvent {
  id: string;
  startTs: number;       // start of the event window (Unix-ms)
  endTs: number;         // end of the event window (Unix-ms)
  label?: string;        // e.g. "turn down"
  pricePerKwh?: number;  // incentive price in ¬£/kWh
  minFlexKwh?: number;   // minimum flexibility the event requires
  maxFlexKwh?: number;   // maximum flexibility the event allows
}
```

Events are loaded from `/data/flexibility_opportunity.json` and parsed by
`mapFlexEvents()`. Time-only start/end values (e.g. `"18:00"`) are treated as
**daily recurring** and expanded into one instance per day within the price-data
range.

### Constraints

- `pricePerKwh` is in **¬£/kWh** (not pence) ‚Äî earnings are converted to pence
  during computation.
- `minFlexKwh` and `maxFlexKwh` define the clamping range. If the event data is
  malformed (`min > max`), `max` is treated as the hard ceiling to avoid
  over-promising.

## computeFlexEarnings

Calculates potential earnings for participating in flex events.

```
computeFlexEarnings(flexEvents, usage, household) ‚Üí FlexEarningResult[]
```

For each event that has a `pricePerKwh`:

1. **Sum** the household's kWh within `[startTs, endTs)` using binary search to
   find the slice boundaries.
2. **Clamp** to `[minFlexKwh, maxFlexKwh]` ‚Äî the effective shiftable amount.
3. **Compute** earnings: `shiftableKwh √ó pricePerKwh √ó 100` (converting ¬£ to
   pence).

Returns a `FlexEarningResult` per qualifying event:

```ts
interface FlexEarningResult {
  event: FlexEvent;
  shiftableKwh: number;  // kWh that could be shifted (clamped)
  earningsPence: number;  // potential earnings in pence
}
```

## computeDailyCost

Sums the total estimated cost in pence for a household over the full usage array.

```
computeDailyCost(usage, prices, household) ‚Üí number
```

For every half-hour row: `cost += kWh √ó lookupPrice(prices, ts)`. This is the
foundation of both `simulateShift` (called twice ‚Äî before and after
modification) and `FlexInsightsPanel` (called once per
[household type](?path=/docs/domain-energy-model--docs) to build the daily cost
comparison).

**Source:** `src/components/Features/FlexInsights/computeFlexInsights.ts`

## sumPeriodUsage

Sums the household's kWh across a set of half-hour timestamps.

```
sumPeriodUsage(usage, household, slotTimestamps) ‚Üí number
```

Used by `simulateShift` to determine the available energy in a period, and by
the `ShiftSimulator` to set the slider maximum.

## simulateShift

Models the cost impact of moving energy between two time periods.

```
simulateShift(usage, prices, household, fromSlots, toSlots, kwhToShift) ‚Üí ShiftResult
```

`fromSlots` and `toSlots` are arrays of half-hour timestamps covering the
source and destination periods (e.g. all 12 slots in "Morning 06:00‚Äì12:00").

1. Computes the **original** daily cost (`computeDailyCost`).
2. Creates a modified usage array:
   - **Removal** is proportional ‚Äî each from-slot loses energy in proportion to
     its share of the period's total usage, preserving the consumption shape.
   - **Addition** is even ‚Äî the shifted kWh are split equally across all to-slots.
   - The total removed is clamped to what the from-period actually has.
3. Computes the **new** daily cost.
4. Returns the delta:

```ts
interface ShiftResult {
  originalCostPence: number;
  newCostPence: number;
  savingPence: number; // originalCostPence ‚àí newCostPence
}
```

A positive `savingPence` means the shift saves money; negative means it costs
more.

## lookupPrice

[Binary-search](?path=/docs/domain-hooks-reference--docs) price lookup used by both
`computeDailyCost` and `computeFlexEarnings`:

1. Find the insertion point for `ts` in the sorted price array.
2. If an exact match exists, return that price.
3. Otherwise fall back to the **preceding** slot's price.
4. If `ts` is before all price data, use the first available price.

## ShiftSimulator state machine

The `ShiftSimulator` component uses a three-state machine driven by two
nullable indices (`fromIdx`, `toIdx`):

### IDLE (`fromIdx = null, toIdx = null`)

- All time-period buttons show the household colour in `outline` variant.
- Slider is disabled, no result is displayed.

### ONE_SELECTED (one of `fromIdx` / `toIdx` is set)

- The selected button renders as `soft` in the household colour.
- The **opposite** row shows hint colours computed by `computeHints`:
  - **Green (success)** ‚Äî shifting here would save money ‚Üí `soft` variant.
  - **Red (error)** ‚Äî shifting here would cost more ‚Üí `ghost` variant.
  - **Grey (mono)** ‚Äî negligible difference ‚Üí disabled.
  - Same slot as selected ‚Üí disabled (can't shift to yourself).
- Slider remains disabled, no result yet.

### COMPLETE (`fromIdx` and `toIdx` both set)

- Both selected buttons render in the **outcome** colour (success/error/mono).
- All unselected buttons revert to `mono outline`.
- Slider is enabled ‚Äî adjusting it live-updates the `ShiftResult`.
- The result area shows: `originalCost ‚Üí newCost` and the saving/extra cost.

Clicking an already-selected button **deselects** it, transitioning back to
ONE_SELECTED or IDLE.

## computeHints

Previews savings for every potential pairing before the user commits:

```
computeHints(usage, prices, household, baseTs, selectedIdx, selectedSide, kwhToShift)
  ‚Üí { color: ButtonColor; disabled: boolean }[]
```

For each time-period slot, runs `simulateShift` against the already-selected
slot and classifies the result:

- **> 0.05p** saving ‚Üí `success` colour, enabled
- **< ‚àí0.05p** (costs more) ‚Üí `error` colour, enabled
- **negligible** difference ‚Üí `mono` colour, disabled
- **same slot** as selected ‚Üí `mono` colour, disabled

The 0.05p threshold (`SAVING_THRESHOLD`) prevents highlighting trivially small
differences.

## Event classification

**Source:** `src/components/Features/FlexInsights/InsightCardList/InsightCardList.tsx`

The `InsightCardList` classifies flex events by their `label` text to determine
the icon, verb, and display order:

- Labels containing **"turn down"** or **"reduce"** ‚Üí `use-less` (down-triangle
  icon, "Use less")
- Labels containing **"turn up"** or **"increase"** ‚Üí `use-more` (up-triangle
  icon, "Use more")
- Everything else ‚Üí `other` (bolt icon, "Flex opportunity")

Cards are sorted: `use-more` first, then `use-less`, then `other`. A "Daily
cost" card always appears first.

## Time groups

**Source:** `src/components/Features/FlexInsights/ShiftSimulator/ShiftSimulator.tsx`

The `ShiftSimulator` divides the day into four 6-hour periods, each covering
12 half-hour slots:

- **Morning** (06:00‚Äì12:00) ‚Äî slots 12‚Äì23
- **Afternoon** (12:00‚Äì18:00) ‚Äî slots 24‚Äì35
- **Peak** (18:00‚Äì00:00) ‚Äî slots 36‚Äì47
- **Night** (00:00‚Äì06:00) ‚Äî slots 0‚Äì11

Each slot index is a half-hour offset from midnight. The `periodSlots()`
helper converts a group into an array of timestamps:
`baseTs + slotIndex √ó HALF_HOUR_MS` for every slot in the range.
