# Flexibility & Shifting

How flex events, earnings, and the shift simulator work.

## FlexEvent

A demand-side flexibility event — a window where the grid operator incentivises
households to reduce or shift their usage.

```ts
interface FlexEvent {
  id: string;
  startTs: number;       // start of window (Unix-ms)
  endTs: number;         // end of window (Unix-ms)
  label?: string;        // e.g. "turn down"
  pricePerKwh?: number;  // incentive price in £/kWh
  minFlexKwh?: number;
  maxFlexKwh?: number;
}
```

Events are loaded from `/data/flexibility_opportunity.json`. Time-only
start/end values (e.g. `"18:00"`) are expanded into one instance per day
within the price-data range. Only events overlapping the usage day (today)
are shown in Flex Insights.

## Flex earnings

`computeFlexEarnings` sums a household's kWh within each event window, clamps
to `[minFlexKwh, maxFlexKwh]`, and computes earnings as
`shiftableKwh × pricePerKwh × 100` (£ → pence). Events without `pricePerKwh`
are skipped.

## Event classification

The `InsightCardList` classifies events by label:

- **"turn down"** / **"reduce"** → use less
- **"turn up"** / **"increase"** → use more
- Everything else → general flex opportunity

A daily cost comparison card always appears first.

## Shift Simulator

Models the cost impact of moving energy between time periods.

### Time groups

The day is split into four 6-hour periods (12 half-hour slots each):

| Period | Range | Slots |
|--------|-------|-------|
| Morning | 06:00–12:00 | 12–23 |
| Afternoon | 12:00–18:00 | 24–35 |
| Peak | 18:00–00:00 | 36–47 |
| Night | 00:00–06:00 | 0–11 |

### simulateShift

```
simulateShift(usage, prices, household, fromSlots, toSlots, kwhToShift) → ShiftResult
```

1. Strips any overlapping timestamps (shifting within the same slot is a no-op).
2. Removes energy from the from-period **proportionally** to each slot's share.
3. Distributes the shifted energy **evenly** across to-period slots.
4. Returns `{ originalCostPence, newCostPence, savingPence }`.

Positive `savingPence` = saves money. Negative = costs more.

### UI behaviour

The simulator has three states driven by `fromIdx` and `toIdx`:

- **Idle** — no selection. All buttons show the household colour as outlines.
- **One selected** — the selected button is filled (`soft`). The opposite row
  shows colour-coded hints: green outline = saves money, red outline = costs
  more, greyed out = negligible or same period.
- **Complete** — both selected. Buttons show the outcome colour. Slider is
  enabled and defaults to the maximum shiftable amount. The same period is
  disabled on the opposite row.
